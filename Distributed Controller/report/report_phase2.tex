\section{Introduction}

This report describes the second stage of the project, where three smart luminaires were integrated into a cooperative distributed illumination system. Compared with the local controller developed in Phase~1, this stage adds CAN-BUS communications, a hub interface between the PC and the control network, a wake-up and calibration procedure executed at start-up, and distributed decision-making algorithms.

The objective is to minimise total energy consumption while keeping desk illuminance above the visibility lower bounds associated with the occupation states LOW and HIGH. Since each luminaire affects not only its own desk but also neighbouring desks, the control problem is coupled and must be solved cooperatively.

% ===================================================================
\section{System Architecture}

The experimental setup is a closed office-scale box with at least three luminaires. Each node contains one RP2040-based microcontroller, one LED driven by PWM, one LDR-based luxmeter, and one MCP2515 CAN controller. The LED and the LDR are mounted vertically so that the sensor measures reflected light on the desk rather than direct light from the source. One node can be connected to the PC through USB serial and operate as a hub, forwarding commands to the CAN network and returning network measurements to the PC.

All nodes run the same firmware. The software is organised as a non-blocking concurrent application. The local lux estimation and control logic run periodically at 100\,Hz, while communications, startup state machines, hub routing, diagnostics, and distributed algorithm iterations are serviced asynchronously. The RP2040 dual-core architecture is exploited as follows:

\begin{itemize}[nosep]
  \item \textbf{Core~0}: control state machines, serial parsing, hub function, calibration logic, and distributed coordination;
  \item \textbf{Core~1}: CAN controller service, interrupt handling, and low-latency frame transfer.
\end{itemize}

This separation keeps the application responsive and avoids blocking interactions between communications and control. The distributed controller computes coordinated actuation targets, while each node still maintains its local illumination and metrics.

\section{CAN-BUS Communication}

The nodes communicate through standard 11-bit CAN frames with fixed-size payloads. The protocol supports direct messages to one node and broadcast messages to all nodes. The main message classes are: node discovery and hello frames, hub commands forwarded from the PC, query/reply messages for the command-line interface, calibration plan messages, and distributed-control data exchanges.

The command interface follows the project tables for duty-cycle setting, lux reference updates, occupancy state changes, enable or disable flags, performance queries, and restart. The hub node translates serial requests from the PC into CAN messages and routes the replies back in the same textual format expected by the command-line interface. This allows the complete network to be operated from a single terminal while preserving the distributed execution of the controllers.

\section{Network Boot \& Calibration}

At wake-up, all nodes enter a discovery phase in which they announce their presence on CAN-BUS and learn which neighbours are active. This stage is required because the boot order is not deterministic. After discovery stabilises, the network starts a synchronous calibration procedure. The manual explicitly requires this procedure to run at every wake-up to compensate for changes in geometry inside the box.

The calibration stage estimates the background illuminance vector $\mathbf{o}$ and the gain matrix $\mathbf{K}$ of the distributed system:

\begin{enumerate}[nosep]
  \item all LEDs off, measuring the external or background illuminance at each desk;
  \item one luminaire turned on at a known duty cycle, the others off, measuring the contribution on all desks;
  \item repetition of the previous step for all luminaires.
\end{enumerate}

The system model is:
\begin{equation}
  \mathbf{L} = \mathbf{K} \cdot \mathbf{d} + \mathbf{o}
  \label{eq:model}
\end{equation}
where $\mathbf{L}$ is the illuminance vector, $\mathbf{d}$ the duty cycle vector, and $\mathbf{o}$ the external illuminance.

The synchronous sequence is executed from a compact calibration plan broadcast on CAN-BUS, so that all nodes follow the same schedule without a central master commanding each slot in real time. This approach reduces traffic, keeps the startup deterministic, and directly provides the self-coupling and cross-coupling gains required by the distributed controller.

\section{Distributed Control Algorithms}

The optimisation problem is:
\begin{equation}
  \min_{\mathbf{d}} \sum_{i=1}^{N} c_i \, d_i
  \quad \text{s.t.} \quad
  \mathbf{K}\mathbf{d} + \mathbf{o} \geq \mathbf{L}_{\text{ref}}, \;
  0 \leq d_i \leq 1
  \label{eq:opt}
\end{equation}
where $c_i$ is the energy cost coefficient for luminaire~$i$.

This formulation captures the two main project goals: satisfy visibility constraints at all desks and minimise the weighted energy cost. Two operating scenarios are relevant in the experiments: identical costs for all luminaires, and different costs to emulate heterogeneous energy penalties.

\subsection{Consensus}

In the consensus-based approach, each node computes a local duty proposal using its row of the gain matrix and the currently shared estimates of the other duties. The proposals are exchanged on CAN-BUS and combined into a common average. Repeating this procedure makes the network converge to a feasible coordinated actuation vector while distributing the computation across the nodes.

\subsection{ADMM}

ADMM was also considered because it separates the optimisation into local primal updates, a projection step onto the illuminance constraint set, and dual-variable updates. This decomposition is attractive for cooperative control because local computations remain simple, while the exchanged variables have a clear physical meaning related to the shared illumination constraints. The penalty parameter $\rho$ controls the convergence trade-off between feasibility and agreement.

The report compares the cooperative solution against the non-coordinated case, where each node acts only on local information and therefore cannot explicitly compensate for cross-coupling.

\section{Results}

The experimental section should focus on the criteria explicitly requested in the manual. First, the wake-up and calibration procedure should be demonstrated with timing information and with the identified $\mathbf{K}$ and $\mathbf{o}$ values. Second, cooperative control should be tested under occupation changes and external disturbances, always comparing coordinated and local-only operation.

For equal energy costs, the expected result is a feasible operating point where all desks satisfy the lower illuminance bounds and the total energy is shared according to the coupling structure. For different energy costs, the cheaper luminaires should naturally carry a larger fraction of the load, provided that the visibility constraints remain satisfied.

The key quantitative metrics are the accumulated energy $E$, the visibility error $V$, and the flicker metric $F$ defined in the project guide. These metrics, together with representative plots of reference, measured illuminance, and control input, should support the final comparison between the distributed method and the non-coordinated baseline.

\section{Conclusions}

The final system satisfies the main objectives of the second stage: asynchronous CAN-BUS communications, hub-based PC interaction, deterministic network wake-up, repeated calibration of self- and cross-coupling gains, and cooperative distributed control. The final discussion should emphasise the practical trade-off between solution quality, communication effort, convergence speed, and implementation complexity.

The most important conclusion to support experimentally is that cooperative control improves the global behaviour of the illumination system when compared with purely local operation, especially under coupling, occupation changes, and external disturbances. The final choice of algorithm should therefore be justified not only by feasibility, but also by measured performance in energy, visibility, flicker, and startup robustness.
